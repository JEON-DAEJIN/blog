## 오늘 배운 내용

데이터베이스는 서비스의 영구적인 기억이라는 것을 배웠다. AI가 저장 코드를 대신 써주더라도, 무엇을 어떤 구조로 기억하게 할지는 결국 사람의 설계와 판단이 결정한다는 점이 인상적이었다. 데이터베이스의 역사는 늘어나는 데이터와 새로운 문제를 해결해온 과정이며, SQL과 NoSQL은 경쟁 기술이 아니라 각자의 상황에 맞는 최적의 도구라는 것도 알게 되었다. 데이터베이스는 결국 표와 관계로 정보를 안전하게 기억하는 시스템이고, AI 시대에는 SQL을 직접 작성하는 능력보다 어떤 데이터를 어떤 구조로 저장하고 연결할지 설계하는 능력이 더 중요하다는 점을 새겼다. 회원가입과 로그인은 사용자를 데이터로 기억하고 안전하게 인증하는 과정이며, 인증은 직접 구현하기보다 검증된 Auth 서비스를 올바르게 활용하는 것이 가장 중요한 실무 역량이라는 것도 배웠다. 그리고 CISA가 특히 강조하는 것 — 복구가 진짜 되는지 가끔 확인하라는 것. 앞서 픽사·GitLab 사고가 보여줬듯, 열어본 적 없는 백업은 백업이 아니다. 오늘은 실제로 ToDo 앱도 만들어보았다.

## 수업내용 요약

### 10. N010. 바이브코딩 데이터베이스

- 데이터베이스란: 프론트(홀)-백엔드(주방) 다음 단계인 '창고' 비유. RAM(단기 기억) vs DB(일기장, 영구 기억)의 차이, 그리고 엑셀과 달리 스키마(규칙)·대용량 검색·동시 접속 처리가 가능한 게 DB의 강점.
- 역사: 1960년대 파일 저장 → 1970년 코드의 관계형 제안 → 1974년 SQL 등장 → 1979년 오라클 상용화 → 2000년대 NoSQL("Not Only SQL") → 오늘날 클라우드. SQL과 NoSQL은 경쟁 기술이 아니라 상황별 최적 도구(폴리글랏 퍼시스턴스).
- SQL과 표로 생각하기: SELECT/INSERT/UPDATE/DELETE = CRUD, 행·열·관계로 표 연결, 1:N vs N:M 관계, 인덱스(책의 찾아보기), ACID(트랜잭션의 4대 원칙), 마이그레이션(DB 구조를 Git처럼 이력 관리).
- 회원가입과 로그인: users 표에 행 추가(가입)·조회(로그인), 비밀번호는 반드시 해시(password_hash)로 저장, 이메일 unique 제약. 세션(서버가 기억) vs JWT(브라우저가 서명된 증명서를 들고 다님) 비교, Redis 캐시와 TTL, httpOnly 쿠키로 XSS 방어.
- 보안과 백업: SQL 주입("Little Bobby Tables" 사례)과 파라미터화 쿼리 방어, 개인정보 최소 수집, 3-2-1 백업 원칙, 픽사·GitLab·Ma.gnolia의 백업 참사 — "복구가 되는지 확인해야 진짜 백업".
- 실습: PRD 작성 → AI가 DB 스키마 제안 → 로컬(SQLite) 또는 클라우드(Supabase+RLS) 선택 → Todo 앱 실제 구현, 검토·승인·Git 커밋으로 되돌릴 지점 확보.

## 인상적인 장면

목·금·토 휴일이 길어서인지, 수업을 듣다 보니 이제 많이 친해진건지 아침 수업 시작 전 채팅이 활발하다. 좀 더 편안한 사이가 되는 게 참 좋다. 오늘은 진진가라는 진실과 거짓을 맞추는 아이스브레이킹 게임을 했는데, 사람들의 거짓말 기술이 갈수록 늘어간다. 오늘은 내가 대상이었는데, 난 거짓말을 못하겠다. 많은 사람들이 정답을 맞춰버렸다.

## 어려웠던 점과 극복 과정

DB 자체의 용도는 알겠으나, SQL 언어를 다 배워야 하는지가 의문이다. 배우지 않고 AI가 다 알아서 해준다고는 하는데, 그래도 어느 정도 알아야 하지 않을까? 세션과 쿠키 개념도 잘 들어오지는 않는다. 쉬는 시간에 다시 한번 체크해봐야겠다. 클로드와 함께 기본적인 것은 배우기로 했다.

## 새롭게 알게 된 점 — 커리어와의 접점

우리 회사 서비스와 관련된 기본적인 서비스 개념조차 챙기지 못했다. 부끄럽다. 다시 뭔가를 만든다면 내가 스스로 체크해볼 수 있는 리스트를 만들고 하나하나 따져보고 물어봐야겠다.

## 기술 질문과 답변

**질문**: 세션 방식과 JWT(토큰) 방식은 "로그인 상태를 어디에 기억하느냐"에서 근본적으로 다릅니다. 오늘 만들어보신 Todo 앱에 만약 로그인 기능을 넣는다면, 세션과 JWT 중 어느 쪽이 더 적합할까요? 그 이유는 무엇인가요?

**답변**: 오늘 만든 Todo 앱이라면 세션 방식(또는 이를 안전하게 제공하는 Supabase Auth)이 가장 적합해. 서비스가 단순하고 보안이 중요하며, 즉시 로그아웃과 계정 차단이 쉬워 초보 프로젝트에 더 안전하고 관리하기 편하기 때문이야.

## 에세이 질문과 답변

**질문**: 오늘 "우리 회사 서비스와 관련된 기본적인 서비스 개념조차 챙기지 못했다"며 부끄러워하셨고, 다음엔 스스로 체크할 수 있는 리스트를 만들어보겠다고 다짐하셨습니다. 오늘 배운 것들(비밀번호 해시, 세션/쿠키, SQL 주입 방어, 백업 원칙) 중에서, 회사의 실제 서비스에 가장 먼저 확인해보고 싶은 항목은 무엇이고 그 이유는 무엇인가요?

**답변**: 우선은 백업이 가장 기본일 것 같고, 그 다음이 세션과 쿠키야. 하지만 모두를 다해야지 어느 하나만으로는 부족하지.

## 한 줄 소감 / 내일의 각오

뭔가 자꾸 만들다 보니 지난번 배운 것들이 복습이 된다. 그리고 클로드와 대화를 통해 뭐든 수정할 수 있다는 자신감이 생겼다.

## 기타 에피소드

ToDo 앱은 사실 내 PC에서 돌아가는데 그것 또한 GitHub에 올려 관리할 수 있음을 알게 되었다. 그리고 개발과정과 코딩은 PRD 형태로 항상 문서로 관리해두면 다시 동일한 것을 개발할 때 좋은 것 같다.

## 클로의 피드백

**좋은 점**
- 기술 질문 답변은 "서비스가 단순하고 즉시 로그아웃·계정 차단이 쉬워야 한다"는 이유를 정확히 짚어, 오늘 자료의 세션 vs JWT 비교표(적합한 곳: 단순한 웹앱, 즉시 로그아웃이 중요할 때)와 정확히 맞아떨어짐.
- 에세이 답변은 백업 → 세션/쿠키 순으로 우선순위를 매기면서도 "어느 하나만으로는 부족하다"고 균형을 잡아, 오늘 배운 여러 원칙이 서로 보완적이라는 점을 스스로 이해하고 있음을 보여줌.

**개선점**
- 기술 답변에 "왜 로그인을 직접 만들지 않고 Supabase Auth 같은 검증된 서비스를 써야 하는지"까지 한 문장 덧붙이면, 오늘 자료의 "인증은 직접 만들지 마세요" 원칙과 더 강하게 연결될 것.
- 에세이 답변에 "왜 백업이 1순위인지"에 대한 구체적 이유(예: 오늘 배운 픽사·GitLab 사례, 또는 회사에서 실제로 겪었던 상황)를 한 문장 추가하면 더 설득력 있을 것.
