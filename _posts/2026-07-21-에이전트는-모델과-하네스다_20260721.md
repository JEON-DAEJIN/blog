## 오늘 배운 내용

어제에 이어 Quest를 제대로 배웠다. 다른 사람의 레포를 fork하고 pull request하는 방법도 알게 되었다. 그리고 에이전트에 대해 본격적으로 알 수 있었고, MCP와 Skill의 개념도 알게 되었다.

## 수업내용 요약

### 11. N011. AI 에이전트 도구

- 에이전트란 무엇인가: 챗봇(어시스턴트)과 에이전트의 차이는 '누가 다음 단계를 정하는가'. 에이전트 = 모델(지능, 운전자) + 하네스(실행 환경, 자동차). 목표를 주면 도구를 스스로 골라 쓰고 결과를 보며 다음 수를 정하기를 반복하는 AI.
- 루프와 ReAct: 계획→실행→관찰→반성을 목표를 이룰 때까지 반복하는 구조(PDCA·OODA와 같은 모양). ReAct(Reasoning+Acting)는 생각-행동-관찰을 번갈아 하며 추측을 검증 가능한 증거로 바꾸는 방식. 2023년 AutoGPT는 완전 자율이 무한 반복·비용 폭증으로 이어진 사례라, 오늘날은 작은 작업 단위·중간 확인을 더 중시함.
- MCP(Model Context Protocol): "AI를 위한 USB-C" — 도구마다 따로 연결하던 것을 표준 규격 하나로 통일. 도구(Tools)·자원(Resources)·프롬프트(Prompts)를 제공하며, 파일시스템·GitHub·Slack·Notion·DB 등에 연결 가능. 연결 규칙일 뿐 안전을 자동 보장하진 않으므로 읽기부터, 쓰기·삭제는 승인 절차를 둘 것.
- 스킬과 규칙 파일: 스킬(Skill)은 반복 작업을 적어 두고 꺼내 쓰는 방법서(SKILL.md), 점진 공개 원리로 평소엔 이름·설명만 훑다가 필요할 때 본문을 읽음. 규칙 파일(AGENTS.md/CLAUDE.md)은 프로젝트 상시 안내문. "새 연결=MCP, 항상 지킬 행동=규칙, 반복 다단계 작업=스킬"로 구분.
- 하네스 종류와 안전: 터미널/편집기/브라우저/클라우드형 네 부류로 나뉘며, 초보는 브라우저형→편집기형→터미널형 순으로 접근이 쉬움. 권한 승인창을 사람들이 약 93% 그냥 승인한다는 조사가 있어, 중요한 폴더에서 자동 수락 금지·되돌릴 지점(git commit) 확보가 안전 습관. CLI+스킬(약 1천 토큰) vs MCP(약 4만 토큰) 비교처럼 혼자 쓰는 작은 프로젝트엔 CLI+스킬이 가볍고, 여럿이 쓰고 권한·기록이 중요하면 MCP가 유리.
- 실습: fork한 레포에 pull request를 보내는 협업 방식을 익혔고, Todo 앱에 CLI(add/list/done/summary)를 붙이고 AGENTS.md·SKILL.md를 직접 만들어 AI가 실제로 CLI를 실행하게 해보는 실습을 진행.

## 인상적인 장면

갈수록 사람들이 진진가(진실 둘, 거짓 하나를 맞히는 아이스브레이킹 게임)에 열심이다. 그만큼 사람들 사이에 관심도 많아지고 동료애를 느끼는 것 같다.

## 어려웠던 점과 극복 과정

MCP 개념이 조금 어렵기는 하다. 그런데 결국 그림을 보니 조금 더 이해가 된다. 이럴 경우 인포그래픽스로 설명해달라고 NotebookLM을 활용해보니 도움이 됐다. 다만 MCP와 Skill, CLI 개념은 여전히 어렵다. 이건 결국 직접 써보고 파악해봐야겠다.

## 새롭게 알게 된 점 — 커리어와의 접점

코드의 리뷰가 아니라 프로젝트 리뷰를 Quest C 형태로 진행했더라면 모든 프로젝트의 성공 가능성을 높일 수 있지 않았을까 하는 생각이 들었다.

## 기술 질문과 답변

**질문**: 오늘 자료(N011)에서는 MCP를 '내 데이터·도구에 새 연결을 붙이는 것', 스킬을 '반복 작업을 외워두는 방법서'로 구분합니다. 오늘 직접 겪어보신(또는 궁금해하신) 상황을 예로 들어, MCP와 CLI+스킬 조합 중 어느 쪽을 언제 쓰는 게 맞다고 생각하시나요?

**답변**: 외부와 연결할 때 MCP를 사용하고 내 안의 작업을 할 때는 주로 CLI와 스킬을 사용하는 것 같기도 한데, 실제로는 대부분 클로드 코드를 사용해서 처리해. 그러나 왜 외부랑 연결하고 어떤 때 사용하는지를 아는 게 중요하기 때문에 항상 개념과 과정은 배워놓으려고 해.

## 에세이 질문과 답변

**질문**: "코드 리뷰가 아니라 프로젝트 리뷰를 Quest C 형태로 했다면 성공 가능성을 높일 수 있지 않았을까"라 적으셨고, 실제로 오늘 이정은님과의 QuestC에서 리뷰의 힘을 느끼셨다고 하셨습니다. 그 리뷰의 어떤 점이 인상적이었나요?

**답변**: 내가 못 본 것을 다른 사람은 볼 수 있다는 게 핵심이라고 생각해. 어떤 경우든 내가 공들인 프로젝트는 좋은 것만 보이거든.

## 한 줄 소감 / 내일의 각오

AI를 배우면 배울수록 내가 뭘 원하는가가 명확해야 하는 것 같다. 그걸 계속 나 스스로 물어보고 발전시켜봐야겠다. 생각하고 내 스스로 머릿속에 그릴 수 있다면 구현하는 것은 어떤 식으로든 가능하다.

## 기타 에피소드

오늘 이정은님과 QuestC를 진행했다. 이 분이 처음 본인의 쇼핑앱은 엉성하게 만들었지만 리뷰만큼은 넘 멋들어지게 해주셨다. 하는 일에 대한 애정이 느껴진다. 프로젝트 끝날 때까지 계속 소통하며 도움을 주고받고 싶어졌다.

## 클로의 피드백

**좋은 점**
- 기술 질문 답변에서 "외부 연결=MCP, 내부 반복 작업=CLI+스킬"이라는 오늘 자료의 핵심 구분(부록 E 비교표)을 정확히 짚었고, 실무에서는 클로드 코드 하나로 대부분 처리하면서도 "왜 이 도구를 쓰는지" 개념은 놓치지 않겠다는 태도는 오늘 자료가 강조한 "도구는 바뀌어도 개념은 남는다"는 메시지와 정확히 맞아떨어짐.
- 에세이 답변 "내가 공들인 프로젝트는 좋은 것만 보인다"는 통찰은, 오늘 메모의 "프로젝트 리뷰를 Quest C 형태로 했더라면"이라는 아이디어의 핵심 근거를 스스로 짚어낸 것 — 왜 제3자 리뷰가 필요한지에 대한 답이 됨.

**개선점**
- 기술 답변에 "실제로 어떤 상황에서 MCP를, 어떤 상황에서 CLI를 써봤는지" 구체적 사례를 한 문장 덧붙이면 오늘 배운 개념과 실전 경험이 더 단단히 연결될 것.
- 에세이 답변에 "이정은님 리뷰에서 구체적으로 어떤 부분을 짚어줬는지" 한 문장을 더하면, "리뷰의 힘"이라는 결론이 더 설득력 있게 뒷받침될 것.
